## Acknowledgements

RepoSage builds on two excellent open-source projects:

- **[GitNexus](https://github.com/abhigyanpatwari/GitNexus)** by [@abhigyanpatwari](https://github.com/abhigyanpatwari) — The knowledge-graph architecture and MCP tool design are directly inspired by GitNexus. Its core idea of "pre-computing structural relationships at index time" (symbol graphs, community detection, execution-flow tracing) showed that this approach was viable.

- **[DeepWiki](https://github.com/AsyncFuncAI/deepwiki-open)** by [@AsyncFuncAI](https://github.com/AsyncFuncAI) — The idea of using an LLM to generate human-readable wiki docs for a codebase, combined with RAG-based Q&A, comes from DeepWiki. Its clean pipeline (clone → vectorize → generate) was a great reference.

Thanks to both projects for their open-source spirit.

## Why This Project

I maintain large mobile codebases day-to-day (Objective-C, Swift, Android Java). These repos are notoriously hard for AI coding assistants: thousands of files, deep inheritance hierarchies, call chains spanning dozens of files — at this level of complexity, AI agents frequently miss dependencies and produce breaking changes.

GitNexus doesn't support ObjC (TypeScript only). DeepWiki only does semantic search (no structural graph). Neither fully solves the mobile problem. So I combined the best of both:

- The **structural precision** of a knowledge graph (who calls whom, who inherits from whom)
- The **semantic understanding** of LLM-generated documentation
- Native support for the core mobile languages: **ObjC, Swift, Java**
- An **MCP server** so AI agents can query the graph directly instead of blindly reading files
- **LLM tasks run through the active Claude session** — no API key required, uses whichever model you're currently running

---

## What It Does

**Code intelligence for ObjC / Swift / Android Java — knowledge graph + MCP server**

RepoSage indexes your mobile codebase into a knowledge graph and exposes it via:

- **Human-readable Markdown wiki** — architecture overview, per-module docs, Mermaid diagrams (generated by Claude, through you)
- **Agent-optimized JSON index** — compressed symbol table, call graph, module topology (no LLM needed, minimal token cost)
- **MCP server** — 11 tools letting Claude Code, Cursor and other AI agents query the knowledge graph directly

> Merges [GitNexus](https://github.com/abhigyanpatwari/GitNexus) (knowledge graph + MCP) and [DeepWiki](https://github.com/AsyncFuncAI/deepwiki-open) (LLM doc generation + RAG Q&A), with native support for Objective-C, Swift and Android Java.

## Quick Start

### One-click Install (Recommended)

The easiest way: hand the install doc to Claude Code and let it do everything automatically.

1. Open Claude Code
2. Say: **"Read this file and follow the steps: /path/to/reposage/SETUP.md"**
3. Claude will clean up old installations, re-index, register the MCP server, verify tool availability, and report back

`SETUP.md` is a self-contained instruction document — Claude reads it and executes all steps end-to-end, no manual commands needed.

---

### Manual Install

**Prerequisites:** Python ≥ 3.10 (3.11 recommended)

```bash
# 1. Clone
git clone https://github.com/your-org/reposage.git
cd reposage

# 2. Create virtualenv (uv recommended)
curl -LsSf https://astral.sh/uv/install.sh | sh
~/.local/bin/uv venv --python 3.11
source .venv/bin/activate

# 3. Install dependencies
uv pip install tree-sitter==0.21.3 tree-sitter-languages tree-sitter-swift \
    mcp chromadb watchdog click rich pyyaml fastapi uvicorn

# 4. Install reposage
pip install -e .

# 5. Add to PATH (persists across shell restarts)
echo '' >> ~/.zshrc  # use ~/.bashrc if you're on bash
echo '# RepoSage CLI' >> ~/.zshrc
echo "export PATH=\"$(pwd)/.venv/bin:\$PATH\"" >> ~/.zshrc
export PATH="$(pwd)/.venv/bin:$PATH"   # take effect immediately
```

## Usage

### Step 1 — Index your repo

```bash
# Single repo
reposage analyze /path/to/your/repo

# Multiple repos at once
reposage analyze /path/to/repo1 /path/to/repo2

# Skip embedding (faster, FTS-only search)
reposage analyze /path/to/your/repo --skip-embed

# Force full re-index
reposage analyze /path/to/your/repo --force
```

Index data is written to a **sibling directory** — the repo itself is untouched:

```
projects/
├── my-repo/                  ← your repo, unchanged
└── RepoSage-my-repo/         ← all generated data lives here
    ├── index.db              ← SQLite knowledge graph
    ├── index.json            ← symbol name lookup table
    ├── symbols.json          ← all symbols (compact format)
    ├── relations.json        ← call / inheritance relations
    ├── modules.yaml          ← module topology
    ├── vectors/              ← ChromaDB vector index
    └── docs/                 ← wiki docs (generated in Step 3)
        ├── ARCHITECTURE.md
        └── modules/*.md
```

### Step 2 — Register the MCP server (once)

```bash
# Recommended: point at a directory, auto-discovers all indexed repos
claude mcp add --scope user reposage -- \
    /path/to/.venv/bin/python -m reposage mcp \
    --repos-dir /path/to/projects

# Or list repos explicitly
claude mcp add --scope user reposage -- \
    /path/to/.venv/bin/python -m reposage mcp \
    --repo /path/to/repo1 \
    --repo /path/to/repo2
```

Restart Claude Code after registering.

### Step 3 — Generate summaries and wiki (LLM phase)

Tell Claude in conversation:

> "Please call get_pending_summaries and get_pending_wiki to complete the LLM generation phase for my-repo"

Claude will automatically loop until done — no further input needed:
1. Fetch a batch of symbols → generate summaries → write back → fetch next batch → repeat until `remaining == 0`
2. Fetch pending modules → generate Markdown wiki → write to `RepoSage-my-repo/docs/`
3. Generate `ARCHITECTURE.md`

Each tool response includes a `next_action` field that instructs Claude to continue immediately without asking for confirmation. This uses **whichever Claude model you're currently running** — no API key needed.

### Step 4 — Ask questions

Restart Claude Code, then query directly in conversation:

```
search("keyboard animation", repo="my-repo")
find_callers("viewDidLoad")
impact("UserService", direction="upstream")
symbol_context("PFBSearchHomeVC")
module_overview("Authentication")
execution_flow("loginButtonTapped")
ask("How is the keyboard animation implemented?")
list_repos()
```

## MCP Tools

All tools accept an optional `repo` parameter to target a specific repository. When only one repo is loaded, it can be omitted.

| Tool | Description |
|------|-------------|
| `list_repos()` | List all loaded repos with stats |
| `search(query, repo?, limit?, language?)` | Hybrid full-text + semantic search |
| `symbol_context(name?, id?, file?, repo?)` | 360° view: callers, callees, parent, doc |
| `find_callers(name, depth?, repo?)` | BFS upstream — who calls this? |
| `impact(target, direction?, depth?, repo?)` | Blast-radius analysis |
| `module_overview(name, repo?)` | Module files, exported symbols, description |
| `execution_flow(entry_point, depth?, repo?)` | Trace execution from an entry point |
| `ask(question, repo?)` | Retrieve context — Claude answers in conversation |
| `get_pending_summaries(batch_size?, language?, repo?)` | Get symbols needing summaries |
| `write_summaries(summaries, repo?)` | Persist generated summaries + re-embed |
| `get_pending_wiki(repo?)` | Get modules needing wiki docs |
| `write_wiki(modules?, architecture?, repo?)` | Persist generated wiki docs |

### `impact` direction values

| Value | Meaning |
|-------|---------|
| `upstream` | Who depends on this? (what breaks if I change it) |
| `downstream` | What does this call? |
| `both` | Both directions |

## How It Works

```
Source files (.m / .h / .swift / .java)
  │
  ▼  tree-sitter AST parsing
  │   ├─ ObjC:  @interface/@implementation, method selectors, message sends
  │   ├─ Swift: class/struct/protocol/func, call_expression chains
  │   └─ Java:  class/interface/method, method_invocation
  │
  ▼  Symbol extraction → SQLite (symbols + relations tables)
  │   ├─ Symbols: id, name, type, file, line, signature, doc_comment
  │   └─ Relations: CALLS / EXTENDS / IMPLEMENTS / CONFORMS_TO / HAS_METHOD
  │
  ▼  Relation resolution (target_name → target_id, cross-file)
  │
  ▼  Module clustering (group by top-level directory)
  │
  ▼  Vector embedding (ChromaDB + all-MiniLM-L6-v2, local, no API key)
  │
  ▼  Agent index generation (RepoSage-<name>/*.json / .yaml)
  │
  ▼  LLM phase via MCP (summaries + wiki, uses your active Claude session)
  │
  ▼  MCP server (stdio JSON-RPC)
       └─ Claude Code / Cursor query the knowledge graph
```

### Language Support

| Language | Extensions | Classes | Methods | Protocols/Interfaces | Inheritance | Call chains |
|----------|-----------|---------|---------|---------------------|-------------|-------------|
| Objective-C | `.m` `.h` `.mm` | ✓ | ✓ (selector) | ✓ (@protocol) | ✓ | ✓ (message send) |
| Swift | `.swift` | ✓ (class/struct/enum) | ✓ | ✓ (protocol) | ✓ | ✓ (chained calls) |
| Java | `.java` | ✓ | ✓ | ✓ (interface) | ✓ | ✓ |

## FAQ

**`No module named 'tree_sitter'`**
```bash
pip install tree-sitter==0.21.3 tree-sitter-languages tree-sitter-swift
```

**`Repository not indexed yet`**
```bash
reposage analyze /path/to/repo
```

**Method names show as `unknown`**

Re-index with `--force`:
```bash
reposage analyze /path/to/repo --force
```

**ChromaDB model download is slow**

`all-MiniLM-L6-v2` (~80 MB) downloads to `~/.cache/chroma/` on first use. Subsequent runs use the cache.

**Adding a new repo later**

Just index it — no need to re-register the MCP server if you used `--repos-dir`:
```bash
reposage analyze /path/to/new-repo
# Restart Claude Code — the new repo is auto-discovered
```

## 为什么做这个项目

我日常维护的是大型移动端代码库（Objective-C、Swift、Android Java）。这类仓库对 AI 编程助手来说出了名的难理解：数千个文件、深层继承体系、跨几十个文件的调用链——在这种复杂度下，AI Agent 经常遗漏依赖关系，导致破坏性改动。

GitNexus 不支持 ObjC（仅 TypeScript），DeepWiki 只有语义搜索（没有结构图谱）——两者都没能完整解决移动端的问题。于是我把两者的优点合在一起，做了 RepoSage：

- 知识图谱的**结构精确性**（谁调用了谁、谁继承了谁）
- LLM 生成文档的**语义理解**
- 对移动端核心语言的原生支持：**ObjC、Swift、Java**
- **MCP 服务器**让 AI Agent 直接查询图谱，而不是盲目读文件
- **LLM 任务通过当前对话的 Claude 执行**——无需配置 API Key，使用你正在运行的任何模型

## 这是什么

**面向 ObjC / Swift / Android Java 仓库的代码智能工具 — 知识图谱 + MCP 服务**

RepoSage 将你的移动端代码库索引为知识图谱，并通过以下方式暴露出来：

## 使用教程

### 第一步：建索引

```bash
# 单个仓库
reposage analyze /path/to/your/repo

# 多个仓库（一次性）
reposage analyze /path/to/repo1 /path/to/repo2

# 跳过向量化（更快，搜索仅用全文）

```

索引数据写在**仓库的同级目录**，仓库本身不会有任何改动：

```

├── my-repo/                  ← 你的仓库，完全不动
└── RepoSage-my-repo/         ← 所有生成内容在这里
    ├── index.db              ← SQLite 知识图谱

    ├── modules.yaml          ← 模块拓扑
    ├── vectors/              ← ChromaDB 向量索引
    └── docs/                 ← Wiki 文档（第三步生成）
        ├── ARCHITECTURE.md
        └── modules/*.md
```

### 第二步：注册 MCP Server（只需一次）

```bash
# 推荐：指定目录，自动发现所有已索引仓库
claude mcp add --scope user reposage -- \
    /path/to/.venv/bin/python -m reposage mcp \
    --repos-dir /path/to/projects

# 或明确指定仓库
claude mcp add --scope user reposage -- \
    /path/to/.venv/bin/python -m reposage mcp \
    --repo /path/to/repo1 \
    --repo /path/to/repo2
```

注册后重启 Claude Code 生效。之后新增仓库只需建索引，无需重新注册（`--repos-dir` 会自动发现）。

### 第三步：生成 Summary 和 Wiki（LLM 阶段）

在 Claude Code 对话中说：

> "请调用 get_pending_summaries 和 get_pending_wiki 工具完成 my-repo 的 LLM 生成阶段"

Claude 会自动循环直到全部完成，无需用户干预：
1. 拉取一批待处理符号 → 生成摘要 → 写回索引 → 拉取下一批 → 循环直到 `remaining == 0`
2. 拉取待处理模块 → 生成 Markdown wiki → 写入 `RepoSage-my-repo/docs/`
3. 生成 `ARCHITECTURE.md`

每个工具的返回结果都包含 `next_action` 字段，指示 Claude 立即继续处理下一批，无需用户确认。这一步使用**你当前对话中的 Claude 模型**，无需配置 API Key。完成后语义搜索质量会显著提升。

重启 Claude Code 后，直接在对话中提问：

## MCP 工具详解

所有工具都支持可选的 `repo` 参数来指定目标仓库。只加载了一个仓库时可省略。

| 工具 | 说明 |
|------|------|
| `list_repos()` | 列出所有已加载仓库及其统计信息 |
| `search(query, repo?, limit?, language?)` | 混合全文 + 语义搜索 |
| `symbol_context(name?, id?, file?, repo?)` | 360° 视图：调用者、被调用者、父类、文档注释 |
| `find_callers(name, depth?, repo?)` | BFS 上游 — 谁调用了这个？ |
| `impact(target, direction?, depth?, repo?)` | 影响范围分析 |
| `module_overview(name, repo?)` | 模块文件、导出符号、描述 |
| `execution_flow(entry_point, depth?, repo?)` | 从入口点追踪执行流 |
| `ask(question, repo?)` | 检索相关上下文 — 由 Claude 在对话中回答 |
| `get_pending_summaries(batch_size?, language?, repo?)` | 获取待生成 summary 的符号批次 |
| `write_summaries(summaries, repo?)` | 写入生成的 summary 并更新向量嵌入 |
| `get_pending_wiki(repo?)` | 获取待生成 wiki 文档的模块 |
| `write_wiki(modules?, architecture?, repo?)` | 写入生成的 wiki 文档 |

### `impact` 的 `direction` 参数

| 值 | 含义 |
|----|------|
| `upstream` | 谁依赖这个？（改了会影响谁） |
| `downstream` | 这个调用了什么？ |
| `both` | 两个方向都分析 |

## 工作原理

```
源码文件（.m / .h / .swift / .java）
  │
  ▼  tree-sitter AST 解析
  │   ├─ ObjC：@interface/@implementation、方法选择器、消息发送
  │   ├─ Swift：class/struct/protocol/func、call_expression 链式调用
  │   └─ Java：class/interface/method、method_invocation
  │
  ▼  符号提取 → SQLite（symbols + relations 表）
  │   ├─ 符号：id、name、type、file、line、signature、doc_comment
  │   └─ 关系：CALLS / EXTENDS / IMPLEMENTS / CONFORMS_TO / HAS_METHOD
  │
  ▼  关系解析（target_name → target_id 跨文件匹配）
  │
  ▼  模块聚类（按顶层目录分组）
  │
  ▼  向量化（ChromaDB + all-MiniLM-L6-v2，本地运行，无需 API Key）
  │
  ▼  Agent 索引生成（RepoSage-<仓库名>/*.json / .yaml）
  │
  ▼  LLM 阶段（通过 MCP 工具，由当前对话的 Claude 执行）
  │   ├─ 为每个符号生成一句话 summary → 写回索引 + 更新向量嵌入
  │   └─ 为每个模块生成 Markdown wiki → 写入 docs/
  │
  ▼  MCP 服务器（stdio JSON-RPC）
       └─ Claude Code / Cursor 直接查询知识图谱
```

### 支持的语言

| 语言 | 扩展名 | 类 | 方法 | 协议/接口 | 继承 | 调用链 |

| Objective-C | `.m` `.h` `.mm` | ✓ | ✓（selector） | ✓（@protocol） | ✓ | ✓（消息发送） |

## 常见问题

**`No module named 'tree_sitter'`**
```bash
pip install tree-sitter==0.21.3 tree-sitter-languages tree-sitter-swift
```

**`Repository not indexed yet`**
```bash
reposage analyze /path/to/repo
```

**方法名显示为 `unknown`**

用 `--force` 重新索引：
```bash
reposage analyze /path/to/repo --force
```

**ChromaDB 向量模型下载很慢**

`all-MiniLM-L6-v2`（约 80MB）在首次使用时下载到 `~/.cache/chroma/`，后续运行直接使用缓存。

**新增仓库后怎么让 Claude 识别？**

只需建索引，无需重新注册 MCP server：
```bash
reposage analyze /path/to/new-repo
# 重启 Claude Code，新仓库自动被发现
```
